# 互联网

是一个通信体系，现在被广泛的成为互联网 / 局域网，这种说法没有严格的限制

# 路由器 / 交换机

互联网通过路由器与交换机，组成了一张交流通信的网络，其中路由器用来组建局域网，交换机则是用来拓展路由器的接口个数

# IP 与 端口号

- IP地址用来标识一个主机 or 位置，本机的IP地址通常是`“127.0.0.1”`，从这可以看出有三个点把IP分为了四部分，每一部分的取值范围是一个字节（0~225），一个主机可以有多个IP（由物理网卡+虚拟网卡决定）

- 端口号是标识主机上的应用程序，每一个使用网络的程序都有对应的端口号，一个进程可以有多个端口号

  端口号是一个16位的数字标识符，取值范围为0到65535，其中0~1023为熟知端口，紧密绑定一些网络服务，我们平时可以用到的端口就是1024 ~ 65535

  除了特定的程序的端口号是固定的，如MySql的默认端口号为3306，http协议的默认端口为80....

# 协议

协议是网络的基本盘，规定了数据通信的格式 / 含义

网络的通信非常复杂，并非只靠一个协议就能解决所有的问题，不同层面需要由不同的协议来规定，故对协议进行了分层——上层协议调用下层协议，或者说下层协议给上层协议提供服务，**<u>这样的好处是1. 解耦合 2. 封装</u>**

1. 解耦合：通信的每一层协议都能够灵活更改替换
2. 封装：上层的协议就不必关心下层的协议，就能直接调用使用，跟Java的思想非常类似

## HTTP

HTTP是处于**<u>应用层的超文本</u>**传输协议

超文本？那跟普通的文本有什么区别呢

- 文本：字符串
- 超文本：HTTP不仅只限于文本传输，还能传输图片、视频、音频、字体，且以二进制的形式来传输

另外，HTTP在传输层中主要依赖TCP的协议	，但HTTP 3.0 版本开始就以UDP协议传输——主要是为了传输效率考虑

那HTTP一般能应用在什么场景呢？

1. web开发中：负责网页前端与后端的通信
2. 移动端APP：负责建立与服务器后端的通信
3. 分布式系统的服务器之间的调用，这种应用近年来逐渐多了起来

   在以前的服务器之间的调用一般都是基于 TCP 或者 UDP + 自定义协议的方式来实现（如protobuf），这是Fiddler中抓包的其中一个网卡内容![image](image-20251012151749-j332jy8.png)

   随着时代变迁，Java版本的微服务兴起，研究出了一套SpringCloud框架应用于服务器之间的调用就直接使用HTTP，实现与SpringMVC的无缝对接，HTTPS也是基于HTTP，引入了“加密层”实现的

了解完了HTTP的历史，那就继续了解HTTP的性质

HTTP是典型的“一问一答”模型，即客户端发出一个请求，服务器就返回一个响应。但其实客户端与服务器之间的交互方式不仅仅只有这一种形式，还有一问多答、多问一答，多问多答的交互模型，以下是它们在应用之间的区别

- 一问一答：HTTP
- 一问多答：下载一个大的文件、流式传输
- 多问多答：远程桌面（toDesk）

另外，HTTP也是不断发展的，比如现在的大模型，大多数都是采用流式传输的形式反馈，流式传输就是基于HTTP实现的“一问多答”的模式

### ？query string

它是查询字符串，也是键值对结构的数据，其中 & 来分割键值对，= 分割键和值

例如上述的`?name=tom&age=20`​就分为了两个键值对分别是`name=tom`​和`age=20`

HTTP中会有很多的键值对，其中就包含query string和header里的键值对，那他们两有什么区分？

- query string 的键值对，是程序员自定义，在开发的时候通过前后端的程序员对接，就能确定某些query string的定义
- header的键值对是HTTP协议里就定义好的

### urlencode

对于query string来说，有时候是需要进行encode转码，因为URL中包含了许多特殊的符号，都是有特定的含义

query string作为URL的一份子，如果在程序员定义的时候也包含了这些特殊符号，可能就会导致浏览器解析失败，无法正确地跳转

所以只要是包含了标点符号或者中英文字符，都要转义

其中中文字符的转义又需要特定封装以下——由于中文字符包含了多个字节，以utf8来讲，如果转义后的字节恰好和某个特殊符号的ascii码值一样就不好了

所以要把转义的内容的每个字节拿出来，使用十六进制标识，同时每个字节都要加上%

## TCP

TCP是处于传输层的传输协议

区别于UDP来说，TCP有着更稳定的传输性，在与服务器发起请求前，会通过三次握手确定连通性

### 三次握手（四次握手）🤝

### TCP如何保证连接的可靠性?

TCP通过<u>**==序号确认和超时重传机制==**</u>确保数据不丢失，借助校验和验证数据完整性，利用<u>==**滑动窗口进行流量控制**==</u>以匹配接收方处理能力，并通过慢启动、拥塞避免等算法动态调整发送速率防止网络过载，最后以**<u>==三次握手建立连接、四次挥手释放连接==</u>**来管理传输通道，从而在不可靠的IP网络上实现可靠的数据传输。

# 封装 / 分用

网络数据传输的基本流程

## 封装

以微信发一条“hi”来说，简略的来说经过了五层封装

1. 应用层

   在键盘上输入“hi”，微信读取了“hi”的内容，以及发送时间，发送方，接收方等数据，并打包了成了一个**<u>==应用数据包==</u>**

   然后再调用传输层的API，把应用数据包传给了传输层
2. 传输层

   传输层大致分为UDP 和 TCP ，以UDP举例，在应用层的数据包基础上再次封装成**<u>==数据层数据包，然后传给网络层==</u>**

   UDP数据包内一般分为报头与载荷，报头保存了双方的端口号，载荷是存放数据的地方

   ![image](image-20250925001101-5l1pwle.png "UDP数据包结构")
3. 网络层

   网络层也有很多协议，以IP协议来说，对数据层数据包进一步封装成**<u>==网络层数据包，传给数据链路层==</u>**

   ![image](image-20250925001355-8bzt6r5.png "IP协议数据包")
4. 数据链路层

   数据链路层也有很多协议，以以太网协议来说，对网络层数据包进一步**<u>==封装为数据链路层数据包，传给物理层==</u>**

   与上面两个不同，以太网协议对数据封装中有帧头与帧尾，整体构成了 **==“以太网数据帧”==**

   ![image](image-20250925001814-1x751jp.png)
5. 物理层

   物理层通过网卡，将数据转换成二进制，通过网线 / 电信号 / 电磁波传播出去了

像以上有报头+载荷封装的数据包，称为报文，是一种特殊格式的字符串

## 分用

分用就是对封装的操作倒置解析

1. 物理层

   接收光电信号，解析成二进制数据，得到数据链路层的数据包，交给数据链路层解析
2. 数据链路层

   解析帧头与帧尾，载荷交给网络层解析

   ![image](image-20250925002248-3fpftfq.png)
3. 网络层

   IP协议解析数据包，对IP报头进一步解析，载荷进一步交给传输层

   ![image](image-20250925002405-n8z02t5.png)
4. 传输层

   UDP协议对报头进一步解析，载荷交给应用层解析

   ![image](image-20250925002459-961ejq0.png)
5. 应用层

   拿到载荷，对内容进行解析，拿到“hi”，发送方，时间等数据

如果传输过程中经过路由器 / 交换机，路由器 / 交换机收到数据也要分用 / 解析，不过只需要解析到网络层即可，需要拿到IP报头的信息（三层解析），交换机相对于路由器只需要解析到数据链路层即可（二层解析），有些先进的交换机也能分用到网络层，功能与路由器接近

# 从输入URL到展示出页面（面试题）

输入URL后先会访问DNS域名解析服务器，~~然后DNS返回一个域名供你访问~~，访问其中会经过IP协议、mac协议和HTTP协议包装发送

- 不是~~DNS返回一个域名供你访问，~~ 而是通常返回一个或者多个IP地址，也会返回CNAME链接等

- 其中IP协议的工作有IP报头，内容中有源IP和目标IP以及载荷，~~大致规划传输路径~~

  - 大致规划传输路径是由**==IP和路由表共同作用==**
  - 其中MTU是数据链路层对于载荷长度的最大限制（MSS）——MSS是TCP层的一个协商值（与MTU相关），故一般IP数据包载荷的拆包分包大多数都受限于MTU，如果数据过大IP数据报可能会被拆包，等到接收方后再分包组包
- mac协议则是更细致的传输线路，~~负责各个路由器之间的传输~~，且路线一般是不固定的，走一步算一步

  - MAC是数据链路层的点对点寻找地址，每一跳的链路地址都会改变；路由是决定“下一跳”，所以走一步算一步是对的，但是MAC本身不参与路由链路地址的选择
- HTTP协议是负责超文本传输和协议的作用，在输入URL时候会自动encodeURL中的一切文本和特殊符号。HTTPS则是一种对HTTP的一种加密

**HTTPS = HTTP + TLS + TCP + IP + MAC**

## 正确解释

- 更正以上的并更加详细的解释

  输入URL -> 浏览器encode解析规范化URL -> DNS解析域名得到IP -> 建立TCP / UDP连接（三次握手） -> 发送HTTP请求报文 -> 服务器解析并返回HTTP响应 -> 浏览器接收，解析HTML、执行JS -> 布局（reflow） -> 绘制 -> 渲染页面最终显示

  1. DNS解析

     首先检查本地的缓存文件，如果没有则向配置的递归解析器查询
  2. 链路层准备

     如果目标IP在同一局域网（LAN）里，主机会把目标IP映射到目标的MAC地址上——路由会收集到和他相连的设备，每个设备的IP，MAC是啥（会广播式地发送ARP数据报，收到ARP请求的设备返回ARP响应，响应中告知自己的IP和MAC）
  3. 建立传输层连接（TCP / UDP / QUIC），以TCP来说

     本地socket发起三次握手

     握手协商MSS（最大报文长度）、窗口大小（流量控制），以及时间戳等

- 从应用层 -> 传输层 -> 网络层 -> 数据链路层 -> 物理层的方向解释

  1. **应用层：** 从浏览器（发送端）输入URL开始，会发送HTTP请求报文

     ![image](image-20251012154833-qnmcx5b.png)

  2. **传输层：HTTP报文会被TCP / UDP封装**

     - 其中加上TCP / UDP报头（源端口、目的端口、校验和，序列号等）
     - 建立连接前通过三次握手
     - 确保可靠传输 / 效率传输

     ![image](image-20251009143529-fvoa50x.png)
  3. **网络层：** 再一步封装IP数据报，决定着“由哪台主机发向哪台主机”

     - 加上IP报头（源IP、目标IP，分片标志等）
  4. **数据链路层：** IP再被封装进帧（以太网 / WIFI）

     - 加上帧头（源MAC、目的MAC，类型）以及帧尾（校验和）
     - 路由决定“下一跳” 操作，每一跳的源MAC和目的MAC都会变化，负责“一跳一跳”传输

       ![image](image-20251009144921-lsk3sde.png)

  5. 物理层：再通过硬件以光纤 / 波的形式传播出去

🧩 数据经过：

1. 浏览器生成HTTP请求
2. TCP可能会拆分成多个段，保证可靠传输（根据MSS决定）
3. IP决定目标地址和路径
4. 网络层通过MAC地址把帧发往路由器
5. 路由器接收到后剥去旧帧头，加上新帧头，转发给下一跳
6. 服务器收到TCP段，解析还原HTTP请求，并发送HTTP响应

💡 注意：

- **MAC 地址**只在同一个链路内有效（例如家里电脑→路由器）
- **IP 地址**在整个互联网范围内有效
- **端口号**区分应用（如 HTTP80、HTTPS443、Spring8080）
