# 遇到的问题

## 不足

1. 主动承认不足：面试时主动说"这个项目当时时间有限，硬编码路径（指配置文件都在本地中）是个技术债，如果重做我会用 Spring Boot 的配置机制
2. 展示改进思路：准备一个"如果给我一周时间优化"的版本（改配置化）
3. 深入一个点：选一个你最有把握的点深入准备（比如 FAISS 原理、倒排索引结构）

## Jieba 分词结果不符合预期

**现象：**  搜"Spring Boot"搜不到，因为分成了"Spring"/"Boot"

**原因：**  Jieba 默认把英文也分了

**解决：**

```java
// 可以加自定义词典
JiebaSegmenter.addWord("Spring Boot");

// 或者预处理，把特殊词用占位符替换
```

**面试说辞：**   
"刚开始用 Jieba 发现搜'Spring Boot'有问题，因为它把英文也切开了。后来了解到可以加自定义词典，也可以对英文做特殊处理。不过项目里 README 大多是中文，这个问题不太严重，就保持原样了。"

## **爬取量大，建立倒排索引时间久**

Gitee 的API有反爬机制，爬的快会限制你访问，跨域问题，爬取量大，建立倒排索引时间久，而且仓库更新速度快，后面就没折腾了。

## Python 服务和 Java 连接不上

**现象：**  Java 调用 `localhost:8080` 报错 Connection Refused

**原因：**

1. Python 服务没启动
2. 端口占用
3. 跨域问题（虽然都是 localhost，但端口不同）

**解决：**

- 先检查 Python 服务是否启动
- 检查端口占用：`netstat -ano | findstr 8080`
- Java 里加超时和重试

**面试说辞：**   
调试的时候发现 Java 调 Python 服务有时候不通。排查发现是启动顺序问题——Python 服务加载模型要几十秒，如果 Java 先启动就会连不上。后来加了健康检查接口，Java 启动前先等 Python ready。

## MySQL 中文乱码

**面试说辞：**

数据库一开始出现中文乱码了，查了是Java里面配置文件忘记加连接字符集了，加上utf-8就没事了。

## TF词频排序简单升级为TF-IDF

当时跟AI去review这个项目的时候，复习AI提了一嘴这个。了解后TF词频排序还是比较基础的，而且只能建索引进行词频统计，TF-IDF可以从保存的文档下手，统计每个词出现的频率，所以添加了全局词频统计，再加上停止词，搜索质量会有明显提升。

# 倒排索引

注意倒排索引本身就是内存结构，不需要额外缓存，如果问到后续优化加Redis时，这里不需要加。

- 传统正向：文档是主体，词是属性

  文档A包含哪些词？文档B包含哪些词？

  doc_1: ["Spring", "Boot", "教程"]  
  doc_2: ["Spring", "MVC", "指南"]  
  doc_3: ["Java", "Boot", "入门"]

搜索"Spring"时：遍历所有文档，检查每个文档的词列表

时间复杂度：O(N × M)  N=文档数，M=每篇文档平均词数

1万篇文档 × 1000个词 = 1000万次比较

- 倒排：词是主体，文档是属性  
  哪些文档包含"Spring"？哪些文档包含"Boot"？

    "Spring": [doc_1, doc_2]  
    "Boot":   [doc_1, doc_3]  
    "Java":   [doc_3]

搜索"Spring"时：直接查哈希表，O(1) 定位

时间复杂度：O(1) 查表 + O(K) 返回结果  K=包含该词的文档数

另外，倒排索引用 **HashMap<String, List<Posting>>**  存储：

- 词项是 String 类型的key，直接哈希定位
- 不用遍历，一次计算就知道存取位置

# 停用词

停用词是指那些被过滤掉的高频但无实际意义的词，比如中文的"的"、"了"、"是"、"在"，英文的"the"、"a"、"is"。

增加停用词是为了过滤掉这些词，减少索引体积，提升搜索精度

eg：

原文：Java的并发编程模型是多线程的

分词后：Java/的/并发/编程/模型/是/多线程/的

不过滤"的/是" → 搜"的"会返回几乎所有文档（无意义）

过滤后 → 索引只存：Java/并发/编程/模型/多线程（更精准）

# FAISS

它负责客户端的“找”

开源的向量相似度搜索库。把文本转成数字向量，然后找"距离最近"的向量。

**通俗理解：**

- 把每段 README 转成 768 维的坐标点
- 用户搜索词也转成坐标点
- FAISS 快速找出"离得最近"的点（最相似的文档）

- 暴力计算（查全表）太慢，FAISS 用索引结构加速这个过程

**为什么不用余弦相似度？**

- L2 距离和余弦相似度在向量归一化后是等价的
- FAISS 对 L2 优化更好

**本项目用的索引类型：IndexFlatL2**

- L2 = 欧几里得距离（直线距离）
- Flat = 暴力搜索，精确但慢（数据量小够用）
- 数据量大时应该换成 IndexIVF 或 IndexHNSW（近似搜索）

# text2vec-base-chinese

它负责爬取仓库README时的“存”，将文档转为数字向量。

是一个中文句子向量模型，把任意中文/英文文本压缩成 768 维的数字向量，让语义相近的文本在向量空间中距离更近

这个模型让搜索引擎能理解意思相近但表述不同的内容

# Jieba

是一种分词工具

**三种模式：**

1. **精确模式**：把句子最精确切开，适合文本分析

   - "我来到北京清华大学" → "我/来到/北京/清华大学"
2. **全模式**：把句子中所有可能的词都扫描出来，速度快但不能解决歧义

   - "我来到北京清华大学" → "我/来到/北京/清华/清华大学/华大/大学"
3. **搜索引擎模式**：在精确模式基础上，对长词再次切分，提高召回率

   - "北京清华大学" → "北京/清华/清华大学/华大/大学"

**本项目选择：搜索引擎模式**（适合搜索场景）
