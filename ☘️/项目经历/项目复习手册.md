> 技术栈：Java/Spring Boot + Python/PyTorch + FAISS + Jieba + MySQL

## 一、核心概念解释

### 1. 倒排索引（Inverted Index）

**什么是倒排索引？**

想象你有一本书（文档），传统的查找方式是翻到第几页看内容（正向索引）。倒排索引就像是书最后的"索引页"：
- 关键词 → 出现在哪些文档的哪些位置

**本项目中的实现：**
```
词项: [文档ID-词频, 文档ID-词频, ...]
"Spring": [(doc_001, 5), (doc_003, 2)]
"Boot":   [(doc_001, 3), (doc_005, 1)]
```

**为什么要用它？**
- 搜索时直接查词项，O(1) 时间找到相关文档
- 不用遍历所有文档，速度快

**本项目的简化处理：**
- 只存了词频（TF），没有存文档位置
- 排序简单按词频总和，没有 TF-IDF 加权

---

### 2. FAISS（Facebook AI Similarity Search）

**什么是 FAISS？**

Facebook 开源的向量相似度搜索库。把文本转成数字向量（embedding），然后找"距离最近"的向量。

**通俗理解：**
- 把每段 README 转成 768 维的坐标点
- 用户搜索词也转成坐标点
- FAISS 快速找出"离得最近"的点（最相似的文档）

**本项目用的索引类型：IndexFlatL2**
- L2 = 欧几里得距离（直线距离）
- Flat = 暴力搜索，精确但慢（数据量小够用）
- 数据量大时应该换成 IndexIVF 或 IndexHNSW（近似搜索）

**为什么不用余弦相似度？**
- L2 距离和余弦相似度在向量归一化后是等价的
- FAISS 对 L2 优化更好

---

### 3. Jieba 分词

**三种模式：**
1. **精确模式**：把句子最精确切开，适合文本分析
   - "我来到北京清华大学" → "我/来到/北京/清华大学"
2. **全模式**：把句子中所有可能的词都扫描出来，速度快但不能解决歧义
   - "我来到北京清华大学" → "我/来到/北京/清华/清华大学/华大/大学"
3. **搜索引擎模式**：在精确模式基础上，对长词再次切分，提高召回率
   - "北京清华大学" → "北京/清华/清华大学/华大/大学"

**本项目选择：搜索引擎模式**（适合搜索场景）

**Jieba 对代码的问题：**
- `getUserById` 会被切成 `get/user/by/id`
- 可能把驼峰命名拆开，导致搜 `getUser` 搜不到

---

### 4. 混合检索策略

**为什么要混合？**

| 方案 | 优点 | 缺点 |
|------|------|------|
| 仅倒排索引 | 精确匹配快、可控 | 不理解语义，"深度学习"搜不到"DL" |
| 仅向量检索 | 理解语义、容错强 | 可能返回无关结果（语义相似但主题不同）|
| **混合** | 兼顾精确和语义 | 复杂度高，可能漏结果（交集太小）|

**本项目的策略：**
- 倒排索引召回候选集（关键词必须匹配）
- 向量检索召回语义相关
- 取交集：既有关键词又有语义相关

**问题：** 如果文档语义很相关但不含关键词，会被过滤掉（这是 trade-off）

## 二、灵魂拷问参考答案

### Q1：为什么选择混合方案？只用一种行不行？

**参考回答：**

"最初我也考虑过只用一种方案。实际测试发现：
- 只用倒排索引，用户搜'深度学习框架'，搜不到只写'DL framework'的仓库
- 只用向量检索，返回的结果有时候太发散，比如搜'苹果'可能返回'苹果手机'也可能返回'水果种植'

混合方案是让倒排索引做'门槛'——必须有匹配的关键词，然后向量检索在候选集里找语义最相关的。这样可控性和语义理解兼顾了。"

**加分补充：**
"不过这个方案有个问题，如果仓库语义很相关但 README 里完全没写关键词，就会被漏掉。如果数据量大，我可能会考虑改成交集+并集加权排序。"

---

### Q2：FAISS 用的什么索引？L2 距离是什么？

**参考回答：**

"用的是 `IndexFlatL2`，就是暴力计算欧几里得距离。因为目前数据量不大，精确搜索够用了。

L2 距离就是两点之间的直线距离，公式是 `sqrt(sum((a-b)^2))`。在 FAISS 里实际用的是平方距离（省去开方），不影响排序。

如果数据量扩大到千万级，我会换成 `IndexIVFFlat` 或 `IndexHNSW`。IVF 是先聚类再搜索，HNSW 是图索引，都是近似搜索，速度更快但有一点点精度损失。"

---

### Q3：Jieba 搜索引擎模式和精确模式的区别？

**参考回答：**

"搜索引擎模式会对长词做额外切分。比如'北京清华大学'，精确模式只切出'北京/清华大学'，搜索引擎模式还会切出'清华/大学/华大'。

搜索场景需要更高的召回率——用户可能搜'清华'也可能搜'清华大学'，所以用搜索引擎模式更合适。

不过 Jieba 对代码处理有问题，比如 `getUserById` 会被切成 `get/user/by/id`。如果专门做代码搜索，可能需要用专门的 code tokenizer。"

---

### Q4：停用词什么时候过滤？构建时还是搜索时？

**参考回答：**

"我目前的实现是搜索时才过滤：
```java
terms.removeAll(stopWords);
```

构建索引时没有过滤。这样做的问题是索引里存了很多停用词（'的'、'了'），浪费内存。

更好的做法应该是构建索引时就过滤，这样索引更小，搜索时也不用再处理。当时做的时候没考虑周全，是以后要优化的地方。"

---

### Q5：为什么要对两个结果集取交集？交集太小怎么办？

**参考回答：**

"取交集是为了确保结果既有关键词匹配又有语义相关。如果不取交集，向量检索可能返回语义沾边但完全不匹配关键词的文档。

交集太小确实是个问题。现在的处理是直接返回空（fallback 到空列表）。更好的策略可能是：
- 交集为空时，降级返回倒排索引的结果
- 或者改成加权排序：关键词匹配度 + 向量相似度综合排序

当时实现比较简单，直接用了交集。"

---

### Q6：倒排索引怎么持久化？ObjectOutputStream 的缺点？

**参考回答：**

"用的是 Java 原生序列化 `ObjectOutputStream`，把整个 `ConcurrentHashMap` 写到文件。

**缺点很明显：**
1. **兼容性差**：如果 `Posting` 类加了字段，旧文件反序列化会失败
2. **体积大**：二进制格式，没有压缩
3. **跨语言难**：只有 Java 能读

**更好的方案：**
- 用 JSON/Protobuf 序列化
- 或者直接用搜索引擎（Elasticsearch、Meilisearch）

当时选择原生序列化是因为快，几行代码就搞定了，适合 demo。"

---

### Q7：硬编码路径怎么处理？

**参考回答：**

"这是项目的一个技术债。当时是快速原型，直接用了绝对路径。如果要部署，我会改成：

1. **Spring Boot 配置化**：把路径写到 `application.yml`，用 `@Value` 注入
2. **环境变量**：敏感配置（模型路径、密码）走环境变量
3. **默认路径**：没有配置时走相对路径（`./data/` 这种）

```yaml
# 改进后的配置
searchengine:
  index:
    path: ${INDEX_PATH:./data/index}
  stopwords:
    path: ${STOPWORDS_PATH:./config/stopwords.txt}
```

这样开发和生产用不同配置，不用改代码。"

### Q8：Python 向量服务挂了怎么办？

**参考回答：**

"现在的代码是 catch 异常返回空列表，用户体验不好。更好的做法是**降级策略**：

```java
try {
    // 调用 Python 服务
} catch (Exception e) {
    log.error("向量服务不可用，降级到仅倒排索引", e);
    return repositoryMapper.selectByIdsInOrder(indexIds);
}
```

还可以加**熔断器**（Resilience4j）：连续失败 N 次后，一段时间内直接走降级逻辑，不再调用 Python 服务。

另外 Python 服务本身应该做**健康检查**，启动时检查模型文件和索引文件是否存在。"

---

### Q9：Gitee API 调用频率限制怎么处理？

**参考回答：**

"现在的代码是 while 循环直接请求，没有延时。如果仓库多，可能会触发限流。

**改进方案：**
1. **添加延时**：每次请求后 sleep 100-200ms
2. **指数退避**：如果被限流（返回 403），等待时间翻倍
3. **缓存结果**：已拉取的仓库信息存本地，避免重复调用

```java
// 简单的延时
Thread.sleep(200);  // 每秒 5 次，比较安全
```

Gitee 文档说认证用户每小时 5000 次，不认证 60 次。我的代码用了 token，但也要控制频率。"

---

### Q10：重启服务会发生什么？怎么优化启动速度？

**参考回答：**

"重启服务需要重新加载倒排索引文件。如果索引很大（GB 级），`ObjectInputStream` 读取会很慢，启动时间可能几分钟。

**优化思路：**
1. **延迟加载**：启动时不加载，第一次搜索时加载（或后台线程异步加载）
2. **内存映射**：用 `MappedByteBuffer` 把文件映射到内存，比 `ObjectInputStream` 快
3. **分片存储**：把大索引拆成多个小文件，按需加载
4. **外部存储**：把索引放到 Redis/Memcached，服务变成无状态

目前项目规模小，没做这些优化。"

---

### Q11：搜索流程有几次 IO 往返？能优化吗？

**参考回答：**

"现在一次搜索的流程：
1. 倒排索引搜索（内存操作，快）
2. HTTP 调用 Python 服务（网络 IO）
3. 求交集（内存操作）
4. 数据库查询（数据库 IO）

总共 2 次外部 IO（Python + MySQL）。

**优化思路：**
- 把向量检索结果缓存，相同查询直接返回
- 如果搜索结果不需要完整 README，可以把仓库信息也缓存到内存/Redis
- 倒排索引和向量检索并行调用（现在不是并行的）"

---

### Q12：`ORDER BY FIELD` 有性能问题吗？

**参考回答：**

"`ORDER BY FIELD(id, 3,1,2)` 会让 MySQL 按指定顺序排序。ID 列表短的时候没问题，如果列表很长（上千个），MySQL 需要做大量比较，性能会下降。

**更好的做法：**
- 在应用层排序：查出结果后用 HashMap 按 ID 映射，再按输入顺序重组
- 或者限定返回数量，比如最多 100 条

```java
// 应用层排序示例
Map<Long, Repository> map = results.stream()
    .collect(Collectors.toMap(Repository::getId, r -> r));
return ids.stream()
    .map(map::get)
    .filter(Objects::nonNull)
    .collect(Collectors.toList());
```

这样把排序压力从数据库转移到应用层，更容易水平扩展。"

---

### Q13：如果加缓存，加在哪一层？

**参考回答：**

"我会加三层缓存：

1. **搜索词缓存**：相同查询直接返回（Redis，TTL 10 分钟）
2. **向量缓存**：Python 服务缓存查询词的向量（本地内存，因为向量计算有 GPU 开销）
3. **文档缓存**：热门仓库信息（Redis，因为仓库信息不常变）

注意倒排索引本身就是内存结构，不需要额外缓存。

缓存穿透、雪崩的处理：
- 空结果也缓存（防穿透）
- 随机 TTL（防雪崩）"

---

### Q14：支持 GitHub 需要改哪些地方？

**参考回答：**

"需要改：
1. **API 接口**：GitHub API 和 Gitee 字段名可能不同（比如 `full_name` vs `fullName`）
2. **认证方式**：GitHub 用 Header 传 token（`Authorization: token xxx`），Gitee 用 query param
3. **Rate limit**：GitHub 限流更严格，需要更严格的频率控制
4. **README 获取**：API 路径不同

**抽象改进：**
可以定义统一的 `GitPlatformService` 接口，Gitee 和 GitHub 各实现一个，用策略模式切换。"

### Q15：中英文搜索问题（搜"深度学习"能搜到"deep learning"吗？）

**参考回答：**

"现在的系统搜不到。因为：
- 倒排索引是精确匹配，词对不上就搜不到
- 向量检索有希望，但 `text2vec-base-chinese` 是中文模型，对英文理解一般

**解决方案：**
1. **多语言模型**：换成 multilingual 模型（如 `paraphrase-multilingual-MiniLM-L12-v2`）
2. **翻译层**：搜索前用翻译 API 把中文转英文，或者建立中英文词表
3. **混合索引**：中文和英文分别建索引，搜索时合并结果

这是跨语言检索（Cross-lingual IR）的问题，需要专门的方案。"

---

### Q16：只能搜 README，能搜代码文件吗？

**参考回答：**

"现在的设计只能搜 README。要搜代码文件：

1. **数据层**：需要拉取代码文件内容，存储量会大很多
2. **分词层**：代码分词和文本不同，需要保留标识符、关键字
3. **索引层**：可能需要单独建代码索引（可以用树结构索引代码的 AST）

如果用向量检索，代码也可以用 CodeBERT 这类模型向量化。

这是一个完整的代码搜索引擎了（类似 Sourcegraph），复杂度比 README 搜索高很多。"

---

### Q17：最不满意的地方？重构先改什么？

**参考回答：**

"最不满意的有三点：

1. **硬编码路径**：这是最快能解决但影响最大的问题，应该优先配置化
2. **缺少测试**：现在没有集成测试，改代码不敢改，应该先补测试
3. **错误处理粗糙**：很多地方直接 catch Exception 返回空，用户得不到反馈

如果重构，我会先：
- 把配置外提，支持环境变量
- 加熔断降级，Python 服务挂了也能正常工作
- 补单元测试和集成测试
- 用 Docker Compose 把整个项目（Java + Python + MySQL）容器化，方便部署"

## 三、Vibe Coding 过程中可能遇到的问题

> 以下问题基于代码反推，面试时可以用来展示你的思考过程

### 问题 1：FAISS 安装失败（Python 环境）

**现象：** `pip install faiss-cpu` 报错，或者安装后 import 失败

**原因：** FAISS 有系统依赖，需要 C++ 编译环境

**解决：**
```bash
# Windows 用 conda 安装更简单
conda install -c pytorch faiss-cpu

# 或者下预编译 wheel
pip install faiss-cpu --no-cache-dir
```

**面试说辞：**
"当时装 FAISS 遇到点问题，Windows 上直接 pip 装不上。查了一下发现需要 C++ 依赖，后来用 conda 装的。也了解到 faiss-gpu 和 faiss-cpu 的区别，我的笔记本没 GPU，就用了 cpu 版。"

---

### 问题 2：Jieba 分词结果不符合预期

**现象：** 搜"Spring Boot"搜不到，因为分成了"Spring"/"Boot"

**原因：** Jieba 默认把英文也分了

**解决：**
```java
// 可以加自定义词典
JiebaSegmenter.addWord("Spring Boot");

// 或者预处理，把特殊词用占位符替换
```

**面试说辞：**
"刚开始用 Jieba 发现搜'Spring Boot'有问题，因为它把英文也切开了。后来了解到可以加自定义词典，也可以对英文做特殊处理。不过项目里 README 大多是中文，这个问题不太严重，就保持原样了。"

---

### 问题 3：Python 服务和 Java 连接不上

**现象：** Java 调用 `localhost:8080` 报错 Connection Refused

**原因：**
1. Python 服务没启动
2. 防火墙/端口占用
3. 跨域问题（虽然都是 localhost，但端口不同）

**解决：**
- 先检查 Python 服务是否启动
- 检查端口占用：`netstat -ano | findstr 8080`
- Java 里加超时和重试

**面试说辞：**
"调试的时候发现 Java 调 Python 服务有时候不通。排查发现是启动顺序问题——Python 服务加载模型要几十秒，如果 Java 先启动就会连不上。后来加了健康检查接口，Java 启动前先等 Python ready。"

---

### 问题 4：索引文件序列化失败

**现象：** 修改 `Posting` 类后，旧索引文件加载报错 `InvalidClassException`

**原因：** Java 序列化有版本号，类结构变了就不兼容

**解决：**
```java
// 添加 serialVersionUID
private static final long serialVersionUID = 1L;
```

**面试说辞：**
"刚开始没加 `serialVersionUID`，后来给 `Posting` 类加了字段，旧索引文件就读不了了。查了资料才知道 Java 序列化有版本兼容问题。后来加了 `serialVersionUID`，并且意识到原生序列化不适合长期存储，应该用 JSON 或 Protobuf。"

---

### 问题 5：MySQL 连接乱码

**现象：** README 内容存入数据库后变成乱码

**原因：** 连接字符集不对

**解决：**
```yaml
# application.yml 里加
url: jdbc:mysql://...?useUnicode=true&characterEncoding=utf8
```

**面试说辞：**
"数据库一开始存中文乱码，查了一下是连接字符集问题。在 JDBC URL 里加了 `useUnicode=true&characterEncoding=utf8` 解决了。也学到了 MySQL 的 charset 和 collation 概念。"

---

### 问题 6：Gitee API 返回格式不统一

**现象：** 有的接口返回数组，有的返回 `{items: [...]}`

**原因：** Gitee API 不同端点返回格式不一致

**解决：**
```java
// 代码里做了兼容处理
if (rootNode.isArray()) {
    itemsNode = rootNode;
} else if (rootNode.has("items")) {
    itemsNode = rootNode.get("items");
}
```

**面试说辞：**
"调 Gitee API 发现不同接口返回格式不一样，有的直接是数组，有的包在 items 里。代码里加了兼容处理，先判断节点类型再解析。这让我意识到对接外部 API 要做容错，不能假设格式固定。"

---

### 问题 7：搜索结果显示顺序不对

**现象：** 明明相关性高的结果排在后面

**原因：** `selectByIdsInOrder` 的 SQL 语法问题，或者交集后顺序丢了

**解决：**
- 检查 MySQL 版本是否支持 `FIELD()` 函数
- 应用层重新排序

**面试说辞：**
"刚开始搜索结果顺序不对，查了一下发现是 `ORDER BY FIELD` 的用法问题。MySQL 5.7 和 8.0 语法有点差别。后来改成应用层排序，虽然多了一步，但更可控。"

---

### 问题 8：内存占用过高

**现象：** 运行一段时间后 Java 进程内存占用很高

**原因：**
- 倒排索引全在内存里
- README 内容也缓存了

**解决：**
- 分页返回结果
- 搜索结果只返回前 100 条
- 截断 README 内容（代码里做了前 100 字符）

**面试说辞：**
"测试时发现内存占用挺高的，因为倒排索引是全内存的。优化了两点：一是搜索结果限制数量，二是返回给前端时 README 只截取了前 100 字符。如果数据量更大，可能需要把索引放 Redis 或 Elasticsearch。"

---

### 问题 9：向量检索和倒排索引结果交集为空

**现象：** 有些查询明明有结果，但返回空

**原因：** 倒排索引召回 100 条，向量检索召回 100 条，交集可能很小

**解决：**
- 增加倒排索引返回数量（从 100 改成 2000）
- 交集为空时降级处理

**面试说辞：**
"调试时发现有些词搜不到结果，但单独用倒排索引能搜到。发现是交集逻辑的问题——倒排返回的 100 条和向量返回的 100 条没有重叠。后来把倒排返回数量增加到 2000，扩大了候选集。也考虑过交集为空时降级返回倒排结果。"

---

### 问题 10：前端页面空白

**现象：** 打开 `localhost:3000` 页面空白

**原因：**
- CORS 配置问题
- 前端没找到后端 API

**解决：**
- 检查 CORS 配置里的 `allowedOrigin` 是否包含前端地址
- 浏览器 F12 看 Network 报错

**面试说辞：**
"前端刚开始调不通，F12 看到 CORS 报错。查了一下是后端 CORS 配置只允许了 localhost:3000，但我用 127.0.0.1 访问的。后来把 127.0.0.1 也加进去了，也学到了 CORS 预检请求的概念。"

## 四、面试技巧

### 1. 诚实比完美更重要

如果被问到不会的问题，直接说：
> "这个我当时没深入研究，如果要做的话我会先查资料/看文档。"

比瞎编要好。

### 2. 主动暴露问题

面试时可以主动说：
> "这个项目还有挺多可以优化的地方，比如硬编码路径、缺少熔断降级。当时主要是快速验证想法，如果要投产我会先改这些。"

这展示你的工程意识。

### 3. 准备改进方案

对每个问题，准备"如果给我时间我怎么改"：
- 硬编码路径 → 配置化 + 环境变量
- 缺少缓存 → Redis 三级缓存
- 单点故障 → Docker Compose 容器化

### 4. 强调学到了什么

即使项目简单，也要强调收获：
- 理解了倒排索引原理
- 学会了 FAISS 向量检索
- 意识到工程化（配置管理、错误处理）的重要性
