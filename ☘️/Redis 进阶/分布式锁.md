# 分布式锁
> 相关笔记：[[缓存|缓存击穿]]、[[Redis/单线程模型|Redis 单线程模型]]

在单机多线程里，我们已经见过“线程安全问题”：多个线程并发执行时，执行先后顺序是不确定的。如果多个线程都在操作同一个共享资源，就需要通过锁来做互斥控制，保证程序在任意执行顺序下，执行逻辑都是 ok 的。

之前学过的锁，比如 Java 的 `synchronized`、C++ 的 `std::mutex`，<mark class="hltr-green-light">本质上都是只能在一个进程内部生效</mark>。它能管住一个 JVM 内的多个线程，但管不住另一台服务器上的另一个进程。之前学的锁都是针对单进程有效的。

到了分布式系统里，就是多进程的问题了：系统中有很多服务实例，每个服务器上的服务都是独立进程。如果这些多进程都要访问同一个公共资源，比如同一个热点缓存、同一条库存数据、同一份订单状态，那么它们之间的执行顺序也是不确定的，也会出现线程安全的问题。<mark class="hltr-orange">所以这里才引入“分布式锁”的解决方案。它解决的不是单个进程内部的线程互斥，而是**多个进程之间的互斥**。</mark>

## 场景引入

比如在缓存击穿里，热点 key 失效后，大量请求同时进来。如果服务只部署一台机器，还可以用本地锁限制这些线程；但真实项目里通常会部署多个服务实例：
1. 请求 A 进入服务实例 1
2. 请求 B 进入服务实例 2
3. 请求 C 进入服务实例 3
4. 每个实例都发现 Redis 没有命中
5. 如果没有分布式锁，它们都会去查 MySQL

这时普通锁就没办法了，因为每个服务实例都有自己的进程空间，它们互相看不到对方进程里的锁。故利用 redis 来做分布式锁，每个进程操作资源时都需要访问 redis 中对应的 key-value，若 redis 中有对应的键值对，则会对这个键值对进行加锁处理，等运行完毕后对这个键值对进行删除（相当于解锁的操作）

![[assets/redis-distributed-lock-flow.svg|719]]

图里要看的是：多个服务实例不会再各自用本地锁，而是都去 Redis 抢同一个 `lock key`。谁抢到这个 key，谁就有资格继续访问 MySQL；没抢到的实例就需要等待、重试，或者直接返回结果。

<mark class="hltr-green-light">这里的 Redis 不是拿来保存业务数据，而是用一个 key 记录“现在谁有资格操作这份资源”。所以分布式锁本质上不是锁住线程，而是锁住某个资源的操作资格。</mark>

## SET NX 操作

回到上面的场景，多个服务实例需要抢同一把锁。那这把锁应该放在哪里呢？只要有一个东西能被所有服务实例同时访问，并且同一时间只能有一个请求把它设置成功，就可以用来表示锁。Redis 的 SET NX 操作就能保证，redis 的事务虽然没有 MySQL 的功能那么强大，但是它的原子性能保证一个命令一定会执行，要么成功，要么失败。

```bash
SETNX lock:product:1001 request-1
```

意思是：只有当 key 不存在时，才设置成功。放到锁的场景里就很直观，`lock:product:1001` 就是锁名，`request-1` 就是当前请求写进去的值。加锁时（就是相当于 set 一个 key，后面加上 nx 则代表如果没有则设置成功，key 原来存在则设置失败，设置失败就代表拿不到那把锁了，相反就可以～）

- 返回 `1`：说明之前没人加锁，当前请求拿到锁
- 返回 `0`：说明锁已经存在，当前请求没拿到锁

多个请求一起抢锁时，第一个请求把 `lock key` 设置成功，后面的请求再执行 `SETNX` 就会失败。这样就能把“多个服务实例一起查 MySQL”的场景，变成“只有一个服务实例能继续查 MySQL”。

🔒 有加锁就要有解锁。释放分布式锁的本质其实就是删除 Redis 里的这个 key：

```bash
DEL lock:product:1001
```

在 Java 里，释放锁一般要放到 `finally` 里。因为业务代码可能成功，也可能抛异常，但只要当前请求已经拿到了锁，最后都应该尽量把锁释放掉。

```java
try {
    // 执行业务逻辑
} finally {
    // 释放锁
}
```

但这里还有一个问题：如果服务不是业务异常，而是 Redis 直接宕机了，`finally` 没有机会执行。<mark class="hltr-blue">这个锁就会一直留在 Redis 里，后面的请求都拿不到锁，这是个非常严重的问题，于是下一步就要引入过期时间。</mark>

## 引入过期时间

给锁加过期时间，是为了避免服务挂掉后锁一直留在 Redis 里。这里的过期时间不是正常释放锁的主要手段，正常情况下还是业务执行完主动 `DEL`。过期时间更像是兜底：万一服务挂了，锁还能自己消失。

早期可能会写成两条命令：

```bash
SETNX lock:product:1001 request-1 # 加锁操作
EXPIRE lock:product:1001 30   # 设置过期时间
```

问题在于这两条命令不是一个整体。如果 `SETNX` 成功后服务刚好挂了，还没来得及执行 `EXPIRE`，锁就没有过期时间了。也就是说，原本是为了解决死锁，结果还是可能留下一个永远不会过期的锁。

所以实际更常用的是把加锁和过期时间放到同一条命令里：

```bash
SET lock:product:1001 request-1 NX EX 30
```


到这里，死锁问题基本能处理了。但新的问题又来了：<mark class="hltr-green-light">锁会自动过期，如果业务执行时间超过 30 秒，会发生什么？</mark>

## 校验 id

现在锁有过期时间了，但释放锁又变得微妙起来。

假设请求 A 拿到锁，过期时间是 30 秒，但业务执行了 40 秒。
1. 第 30 秒，锁自动过期
2. 请求 B 拿到同一个 lock key
3. 第 40 秒，请求 A 执行结束，准备释放锁

如果请求 A 直接 `DEL lock:product:1001`，就会把请求 B 正在拿的锁给删掉，这个就有问题了。所以加锁时不能只写一个随便的 value，而要写当前请求自己的 id。这个 value 不是给业务看的，是给释放锁时校验身份用的。

释放锁时先判断：
1. Redis 里的 value 是不是我的 id（客户端自己的 ID）
2. 是我的，才能删除
3. 不是我的，不能删

这个 id 可以是 UUID，也可以是业务里能保证唯一的请求标识。核心是让锁带上“签名”：谁加的锁，谁才能释放。

## Lua 脚本

校验 id 之后，看起来可以这样写：

```bash
GET lock:product:1001
DEL lock:product:1001
```

但这里又有一个问题：`GET` 和 `DEL` 分开执行，中间可能插入其他请求。要注意：一个服务器内部，是有可能多线程的。此时就有可能一台服务器有两个线程都在执行上述的解锁操作：比如请求 A 执行 `GET` 时发现 value 是自己的，但还没来得及 `DEL`，锁过期了，请求 B 又拿到了锁。此时请求 A 执行 `DEL`，还是会删掉请求 B 的锁。请求 B 这时再执行  `DEL`。这听上去觉得没什么，不就是删除多一次嘛，反正锁最后都是释放的。

实则不然！如果此时引入另一个服务器执行加锁，就有可能出问题了，比如：
1. <mark class="hltr-pink">在请求 A 执行 `DEL` 之后，请求 B 执行 `DEL` 之前</mark>，服务器 2 的请求 C 正好要执行加锁操作 `SET NX`，因为请求 A 此时已经解锁了，请求 C 是可以拿到这把锁的
2. 紧接着，请求 B 又要执行 `DEL`，就把服务器 2 的请求 C 刚拿到的加锁操作给解锁了

所以归根结底，还是因为 `GET` 和 `DEL` 不是原子产生的问题。虽然 Redis 的事务也能解决这个问题，但是实战中有更好的解决方案：把“判断 id”和“删除 key”放到同一个 Lua 脚本里，让 Redis 一次执行完：

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

这个脚本的含义是：
1. 读取 lock key 的 value
2. 判断它是否等于当前服务器的 id
3. 如果相等，就删除锁
4. 如果不相等，直接返回 `0`

Redis 执行 Lua 脚本时，中间不会插入其他命令，所以这个释放过程是一个整体。这里就把“判断 id”和“删除锁”合在了一起，也就处理掉了“误删别人锁”的问题。

到这里，一个比较完整的单 Redis 节点锁就出来了：
1. `SET key value NX EX seconds` 加锁
2. value 存当前请求自己的 id
3. 业务执行完，用 Lua 脚本校验 id 后释放锁

## 看门狗（Watch Dog）

引入过期时间后，还有一个比较重要的问题：业务到底要执行多久，有时候不好估量。

如果过期时间设置得太短，业务还没执行完，锁就过期（释放）了；如果过期时间设置得太长，其他请求要等很久。

这就是看门狗要解决的问题：锁不能无限期存在，但业务没执行完时，也不能让锁提前过期。

看门狗的思路是：
1. 加锁时不手动指定超时时间
2. 给锁一个默认过期时间，比如 `lockWatchdogTimeout = 3000ms`
3. 当进程还在执行着，业务还没结束，但是已经过了默认的过期时间，看门狗就会继续给锁续上过期时间（比如 1 s）
4. 如果 1 s 过后业务还在执行中，就会再续一次（业务未停就无限续杯）
5. 等到业务执行完后，就会主动释放锁，看门狗也就停了

可以简单理解为：过期时间负责兜底，看门狗负责在业务还没结束时帮锁续命。这样就不用完全靠自己猜业务要跑多久。

注意，如果服务已经挂了，看门狗不会继续续期。锁最终还是会过期，后面的请求还有机会拿到锁。也就是说，看门狗解决的是“业务还活着但执行时间不好估”的问题，不是让锁永远不释放。

## RedLock 算法

前面的方案主要是在一个 Redis 实例上加锁。再往下看，就会发现问题从“业务服务会不会挂”变成了“Redis 自己会不会挂”。

一个典型场景是主从复制：
1. 请求 A 在 Redis 主节点上加锁成功
2. 锁数据还没同步到从节点，主节点宕机
3. 从节点升为新的主节点（从节点同步数据有延迟，可能此时主节点的数据未同步至从节点）
4. 导致请求 B 又在新的主节点上加锁成功

这样就会出现两个请求都以为自己拿到了锁的问题。这个问题不是业务代码里的 `DEL` 能解决的，而是 Redis 节点层面的故障导致的。

RedLock 的思路是：不要只依赖一个 Redis 节点，而是准备多个互相独立的 Redis 主节点，每个主节点后面又是一套主从复制的结构。客户端加锁时，向多个节点都尝试写入同一个 key 和同一个 id。如图
![[Pasted image 20260608235559.png|537]]

以 5 个 Redis 节点为例：
1. 客户端记录开始时间
2. 向 5 个 Redis 节点尝试加锁
3. 如果在有效时间内，至少 3 个节点加锁成功，就认为拿到了锁
4. 如果没达到多数，或者耗时已经超过有效时间，就认为失败
5. 失败时，要把已经拿到的那些锁也释放掉

这里的“至少 3 个”就是多数派。它的意思是：不能因为某一个 Redis 节点出问题，就让两个客户端同时认为自己拿到了锁。RedLock 它强调的是多个独立 Redis 主节点，通过多数派来判断锁是否成立。

解锁的时候也是同理，需要向多个 redis 主节点发送通知

## 记忆链路
可以按这条链路来记：

1. `SETNX`：解决“只有一个请求能加锁”的问题
2. `NX EX`：把加锁和过期时间放到一条命令里，避免锁一直留着
3. `requestId`：给锁加签名，避免释放别人的锁
4. Lua 脚本：把判断和删除放到一个整体里执行
5. 看门狗：业务时间不好估时，自动续期
6. RedLock：Redis 节点本身也可能故障时，用多个独立节点投票
